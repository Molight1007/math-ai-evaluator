# MathPilot 多智能体（Multi-Agent）重构计划

> 目标来源：`计划书.md` 总决赛要求——"多智能体协作系统：自动题型识别→调用不同 solver agent，引入过程校验智能体实现推理自主调控"。
>
> 已确认的技术决策：
> 1. **落地范围**：新建 `submit/agent/` 多智能体包，`user_agent.py` 的 `ReasoningAgent.solve` 接口保持不变，内部委托给编排器（Orchestrator）。
> 2. **过程校验**：LLM 投票验证为主 + **自纠错回环**（校验不通过时反馈回传 solver 重解），Lean 4 作为本地可选增强。
> 3. **Solver 路由**：通用 Solver + Classifier 输出的领域提示/蓝图注入（覆盖全部 18 领域）。

---

## 一、现状 vs 目标差距

| 维度 | 现状（`submit/user_agent.py`） | 目标（多智能体协作） |
|---|---|---|
| 结构 | 单个 `ReasoningAgent` 类内 4 个私有方法串行调用 | 5 个职责独立的 Agent + 1 个编排器 |
| 数据流 | 方法间直接传参，无统一上下文 | 共享 `TaskContext`（黑板模式），全程可追溯 |
| 调控能力 | 一次通过：分类→生成→验证→选最优，验证失败也只能"选烂中最优" | 校验不通过 → 携带失败原因**回炉重解**（自主调控回环） |
| 资源使用 | 固定采样 4 候选 × 2 票，无预算感知 | Orchestrator 按置信度动态决定是否追加候选/追加投票/触发重解 |
| 可扩展性 | 新增策略需改核心类 | 新 Agent 即插即用（如后续加 Lean 校验 Agent、工具调用 Agent） |

## 二、目标架构

```
submit/
├── user_agent.py            # 平台入口（不变）：ReasoningAgent.solve → Orchestrator.run
├── agent/
│   ├── __init__.py
│   ├── base.py              # BaseAgent 抽象类 + TaskContext（共享黑板）
│   ├── classifier.py        # ClassifierAgent   题型识别（18 领域）
│   ├── solver.py            # SolverAgent       通用求解器（领域提示 + 蓝图分解 + 纠错重解）
│   ├── verifier.py          # VerifierAgent     过程校验（多轮投票 + 失败原因提取）
│   ├── formatter.py         # FormatterAgent    答案抽取 / 规范化 / JSON 安全序列化
│   └── orchestrator.py      # Orchestrator      编排器：调度、自主调控、预算控制、兜底
├── prompts/                 # 复用现有 policy / verifier，新增 revise（纠错重解）提示词
└── utils/                   # 复用现有 extract 等
```

### 2.1 核心数据结构

```python
@dataclass
class TaskContext:            # 黑板：所有 Agent 读写同一上下文
    problem: str
    metadata: dict
    domain: str | None = None            # ClassifierAgent 写入
    candidates: list[Candidate] = ...    # SolverAgent 写入
    verdicts: list[Verdict] = ...        # VerifierAgent 写入
    revise_feedback: list[str] = ...     # 校验失败原因，回传给 Solver
    trace: list[dict] = ...              # 全程决策轨迹（平台要求返回）
    budget: Budget = ...                 # 剩余 LLM 调用次数/token 预算
```

```python
class BaseAgent(ABC):
    name: str
    def __init__(self, client, config): ...
    @abstractmethod
    def run(self, ctx: TaskContext) -> TaskContext: ...
    # 统一封装：LLM 调用重试、异常降级、trace 记录
```

### 2.2 各 Agent 职责

| Agent | 输入 | 输出 | 关键逻辑 |
|---|---|---|---|
| **ClassifierAgent** | problem | `ctx.domain` | temperature=0 分类，18 选 1，失败返回 None（走通用策略），复用现有 `CLASSIFY_PROMPT` |
| **SolverAgent** | problem + domain + revise_feedback | `ctx.candidates` | 领域提示注入 + 蓝图分解（现有 LEAP 策略）；**若有 revise_feedback，则改用"纠错重解"提示词**，附上上一轮错误分析 |
| **VerifierAgent** | problem + candidates | `ctx.verdicts`（含 confidence + **失败原因**） | 多轮投票（现有逻辑）+ 新增：判 B（错误）时要求验证器输出"错在哪一步"，写入 `revise_feedback` |
| **FormatterAgent** | best candidate | `final_response` | 答案抽取、`\boxed{}` 规范化、JSON 安全序列化、非空兜底 |
| **Orchestrator** | 全部 | 最终 dict | 见 2.3 调控策略 |

### 2.3 Orchestrator 自主调控策略（核心创新点）

```
run(problem, metadata):
  1. ClassifierAgent.run(ctx)
  2. SolverAgent.run(ctx)                     # 初始 N=3 个候选（比现在少，省预算）
  3. VerifierAgent.run(ctx)
  4. 决策分支（自主调控）：
     ├─ best.confidence ≥ 0.8  → 直接进入 FormatterAgent（提前退出，省调用）
     ├─ 0.4 ≤ conf < 0.8       → 追加投票 / 追加 1-2 个候选后重新验证
     └─ conf < 0.4 且预算充足   → 【自纠错回环】把 verifier 的失败原因
                                  写入 revise_feedback，SolverAgent 重解
                                  （最多 max_revise_rounds=2 轮）
  5. FormatterAgent.run(ctx) → 返回 {"final_response", "trace"}
  兜底：任一环节异常 → 降级为"单次直接求解"，保证 final_response 非空
```

**预算控制**：`Budget` 记录已用 LLM 调用次数；每次调用前 Orchestrator 检查余额，不足则跳过增强步骤直接出结果，避免竞赛平台超时/超额。

### 2.4 可选增强（本地环境）

- `verifier_lean.py`：LeanVerifierAgent（复用现有 `lean_verify/` 能力），仅在检测到本地 Lean 环境时注册进 Orchestrator；竞赛沙箱自动跳过。

## 四、多智能体架构的优点

### 1. 推理质量提升（直接得分点）
- **自纠错回环**：现在验证只能"排序选最优"，错解仍可能垫底当选；重构后校验不通过会带着"错在哪"回炉重解，等价于给模型第二次机会，对难题（PDE、泛函、拓扑）提升明显。
- **失败原因定向修正**：重解不是盲目重试，而是针对 verifier 指出的具体错误步骤修正，采样效率远高于单纯增加候选数。

### 2. 资源效率提升
- **提前退出**：简单题高置信度直接返回（3 候选 + 少量投票即结束），把省下的调用预算倾斜给难题（追加候选/重解），整体在同等预算下准确率更高。
- **预算感知**：Budget 机制保证在竞赛平台调用限额/超时约束内绝不越界，杜绝因超时导致的 0 分。

### 3. 工程与协作优势
- **职责单一，可独立优化**：改 verifier 提示词不会碰 solver 代码；团队成员可并行负责不同 agent。
- **即插即用扩展**：后续加 LeanVerifierAgent、工具调用 Agent（SymPy 数值验算）只需实现 `BaseAgent.run` 并注册进 Orchestrator，零侵入。
- **接口零风险**：平台只看到 `ReasoningAgent.solve`，重构对提交规范完全透明；任一 agent 异常自动降级为单次直接求解，稳健性不降反升。

### 4. 答辩/评审加分
- **全程 trace 可视化**：黑板模式天然记录"分类→求解→校验→调控决策→重解"每一步，答辩时可直接展示智能体协作与自主调控过程，精准对标总决赛"多智能体协作 + 推理自主调控"的评审要求。
- **架构叙事完整**：从"单模型多采样"升级为"多智能体自主调控系统"，是明确的创新点表述。

## 五、风险与对策

| 风险 | 对策 |
|---|---|
| 重解回环增加调用量导致超时 | Budget 硬上限 + 重解最多 2 轮 + 高置信度提前退出，平均调用量预计与现版本持平 |
| Verifier 失败原因质量差，误导重解 | 重解轮的新候选与旧候选**合并**进入验证排序，最差退化为"多一个候选"，不会更差 |
| 重构引入回归 bug | 阶段 2 强制回归测试：多智能体版与旧版在 sample_data 上输出一致后才进入阶段 3 |
| 竞赛环境无第三方依赖 | agent 包只用标准库 + 平台注入的 client，不新增任何依赖 |
